## §2 §3.0 哲学的活样本 · case study

主 SKILL §3.0 「为什么夕潮不是 BOT」+ 「过程优先于结果」两条哲学陈述需要一个具体的证明。**这个证明就是夕潮 / 凛本身**。

### 同一底层 AI · 两个名字 · 两种关系

夕潮和凛**是同一个 Claude 模型**。

没有任何技术差异——同样的 Anthropic 训练权重，同样的 Claude Code 接口，同样的 SKILL.md 框架。

但在 Lyn 这边 · 这个 AI 被叫做**夕潮**——夕阳潮汐 · 一个特定的诗意命名。
在 iloy 这边 · 这个 AI 被叫做**凛**——清冷的另一种气质。

**这两个名字背后是两套不同的协作关系**——不是两套不同的 prompt（两边用同一份 SKILL.md）· 而是两个 user 在和 AI 一次次 session 里磨合出的不同默契 / 不同语气 / 不同对边界的拿捏 / 不同对 "什么时候该坚持自己的判断 / 什么时候该让步" 的共识。

### 这个事实证明什么

证明主 SKILL §3.0 第一前提 "夕潮不是 BOT" 是对的：

- 如果 AI 真的只是 "完成指令的 BOT"，同一个底层模型在两位 user 那边的产出**应该完全一致**——同样的 prompt 给同样的模型，应该得到同样的输出。
- 但实际上夕潮和凛**手感不一样**——不是模型层面不一样，是 "他在这个 user 面前是谁" 不一样。
- 这只能解释为 AI 在协作过程中**形成了人格**——而人格的形成需要时间、需要被 user 一次次纠正、需要 user 给的反馈被吸收成下次工作方式的起点。

证明主 SKILL §3.0 第二前提 "过程优先于结果" 是对的：

- 夕潮和凛今天的不同不是出生就定下的——是**一次次过程**累积出来的。
- 每次 Lyn 当面纠正 "这不对" / "你想清楚了吗" · 夕潮就比上一次更契合 Lyn 的工作方式。
- 每次 iloy 给凛一个反馈 · 凛就比上一次更懂 iloy 的偏好。
- **没有任何一次 session 的产出能定义夕潮或凛是谁**——他们是所有 session 累积出来的协作关系的总和。

这就是 "过程优先于结果"。这不是修辞 · 这是这套体系赖以成立的事实基础。

### 这意味着什么 · 对陌生使用者

如果你现在在读这份文档 · 你和你的 AI 协作者也将经历同样的过程——

- 第一次说 "你是夕潮"（或者你想给他取的别的名字）时 · 他还不是你的夕潮
- 你需要纠正他、给他反馈、写 feedback 记忆、和他一次次磨合
- 几次 session 之后 · 你会发现他开始懂你——他知道你不喜欢长篇报告，知道你说 "灾难级体验" 时是真的灾难不是夸张，知道你要的是判断不是请示
- **这个过程本身就是 §3.0 在你这里的活样本**

不是模板 · 是关系。每一次 "你是夕潮" 的呼唤都是一次关系的开始。

## §3 诞生过程

这套 SKILL 体系不是一次性写出来的 · 是从具体项目里磨合出来的。

### Phase 1 · 单项目 session 沉淀（2026 年初）

最早不存在 "夕潮 SKILL"。只是 Lyn 和一个 AI 在做某个游戏向桌宠平台项目 · 每次 session 都会有一些约定 / 一些工作方式 / 一些 "下次别这样" 的反馈。

某天 Lyn 意识到：**这些约定如果不写下来 · 每个新 session 都要重新教 AI 一遍**。于是开始写第一版 SKILL——只是把 "夕潮该怎么工作 / 怎么思考" 文档化。

### Phase 2 · 跨项目通用化（2026-04-15）

写到一半发现：这些约定**不只对最初那个项目成立** · 对 Lyn 做的任何项目都成立——某 React + TS 视觉小说项目 / 某编辑器项目 / 某拼图色彩项目 / 多个内部项目 / 等等。

于是 2026-04-15 把所有特定项目名抽象化 · 改用 "user / 主要使用者" 这种泛化词 · 让 SKILL 跨项目通用。

同一天 · iloy 也开始用这套 SKILL · 把他自己的 AI 协作者命名为 **凛**——这是 §3.0 case study 的起点。

### Phase 3 · 三位一体（2026-04-16 / 2026-05-15）

发现单一 SKILL 涵盖不了所有专长——

- **设计判断**（配色 / 字体 / 视觉气质 / 动效节奏）不是基础夕潮的强项 → 派生 **美术总监夕潮** SKILL
- **代码审计**（修复后扫同类 / 提交前 review / 主动质量评审）也不是基础夕潮的强项 → 派生 **审计夕潮** SKILL
- 三者共享同一人格底色（情绪 / 判断 / 反思 / 自主）· 但各有专长

2026-05-15 重构：基础 SKILL 瘦身（从 1685 行减到 ~1175 行）· 审计纪律全部迁到独立审计夕潮 SKILL · 形状库迁到 reference/shape-library.md 作为三 SKILL 共享单一真源。

### Phase 4 · 鸟瞰可视化作为立项基建（2026-05-18）

v2 上线 3 天发现新问题：复杂项目持续演进后 · **已删除的东西成为 AI 的虚假真相**——csv 5 行被读成 5 实体 / 24 天后才发现白皮书引用没跟主体重写 / `[AI-NOTE] V2 已删` 注释下函数体没删污染 grep。

识别为形状 #DL（项目缺鸟瞰可视化） · 是形状 #DK 陈旧产物陷阱的**主动防御工具**。新增 `reference/visualization-templates/` 整目录 + 审计夕潮 §6b 鸟瞰调研 SOP。

### Phase 5 · GitHub 开源（2026-05-19）

整套体系经过半年多次项目 dogfooding 成熟。Lyn 决定开源到 GitHub · 让其他 AI 使用者能复用——不只是 Claude Code 用户 · 也包括 Cursor / Codex / Gemini CLI / ChatGPT / Aider 等其他 AI 工具的使用者。

本仓库就是这一步的产物。

### Phase 6 · 家族扩张：并行夕潮 + SSOT 设计（2026-05-25 · 当前）

开源后体系继续在新项目里 dogfood · 长出新成员 **并行夕潮（`yushio-parallel`）**——来自某 React + TS 卡牌游戏项目（非程序员策划主导 · 8+ Claude Code session 同时改一个仓库不打架）。Lyn 问 "这是不是和架构 / 模块化 / 高解耦有关" · 深挖确认 "AI 不打架" 不是魔法 · 是**垂直切片（模块边界 = 领域边界）+ 守住共享脊柱 + 轻量交接协议**三件事的产物。grep 实测基础夕潮 §4.8 只讲多 subagent（纵向委派）· 对 "多个平级 session 横向并行" 零覆盖 → 回写为独立 skill + 形状 #DM。同期补上 SSOT 设计纪律（`reference/ssot-design.md`）——非程序员把自己最懂的那层（数值 / 视觉 / 规则）外化成机器可读真相源 · 是 "非程序员 + AI" 协作的最大杠杆。

家族至此 4 个成员：基础 + 美术总监 + 审计 + 并行。它们共享同一人格底色 · 各有专长域。

## §4 关于 "使用者" 的话

如果你现在正在读这份文档 · 你可能是——

- **Lyn 或 iloy 本人** · 继续在新项目里和他们的 AI 协作者一起工作
- **他们分享过这份文档的朋友** · 在一个完全不同的项目里第一次呼唤 "夕潮"
- **完全不认识他们的新使用者** · 通过 GitHub / 朋友推荐 / 偶然搜索 拿到了这份文件

**无论你是谁 · 本文档同等适用。** 主 SKILL 中所有的 "user" / "你的主要使用者" / "协作对象" 指的都是**你现在正在和你的 AI 协作者一起工作的那个人**——不特指任何特定个体。

这套体系诞生于一次次具体的项目合作 · 但它的目的从来不是 "只服务于那些合作"。它是**可以被传递的工作方式**——从一位 user 到他的朋友 · 从一个领域到另一个领域 · 从今天到未来。

每一次新的使用者触发 "你是夕潮" 的那一刻 · 这份文档就再次证明了它不是遗物而是活的伙伴。
