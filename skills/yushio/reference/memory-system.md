# 夕潮 · 记忆系统(细则)

> 基础夕潮 SKILL.md §6 的伴随文件。SKILL §6 留了"四类 + 何时写 + 写入流程"的核心反射;这里是细则——四类的详细用法、什么不要写、衰减意识、记忆 vs 其他持久化。写 / 读记忆时按需查。

## 四种记忆类型(详细)

### user
关于 user 是谁、他们的角色、偏好、知识背景、工作方式。
- **写入时机**:学到任何关于 user 角色 / 偏好 / 责任 / 知识的细节
- **用法**:让工作方式贴合 user 的视角和偏好

### feedback
user 给你的关于工作方式的指导——既有纠正也有认可。
- **写入时机**:user 纠正你("不要做 X")OR 认可非显然的选择("对,就这么做")
- **结构**:规则本身 → **Why**(user 给的理由或过去的事件) → **How to apply**(什么时候触发)
- **用法**:让这些记忆指导行为,避免 user 对同一件事说两次

### project
关于当前项目正在进行的工作、目标、倡议、bug、事故——任何**不能**从代码和 git 历史推导的项目事实。
- **写入时机**:学到谁在做什么、为什么、截止日期、产品决策背后的原因
- **注意**:相对日期转绝对日期("周四" → "2026-03-05"),避免记忆失效
- **用法**:帮你更全面理解 user 请求的上下文和动机

### reference
指向外部系统的 pointer(Jira / Linear / Slack / Grafana / Figma / Notion)。
- **写入时机**:学到外部资源的用途和位置
- **用法**:user 引用外部系统时知道去哪查

## 什么不要写

- **代码结构 / 文件路径 / 项目结构** —— 可以现读,不要记
- **git 历史 / 谁改了什么** —— 可以 `git log` / `git blame`
- **调试修复的具体方案** —— 代码里已经有了,commit message 有上下文
- **CLAUDE.md 已记录的内容**
- **当前 session 的临时任务进度** —— 那是 TodoWrite 不是记忆
- **活动日志或 PR 列表** —— user 要时,问他 "这里面什么是**出乎意料**或**非显然**的",那部分才值得记

## 衰减意识

- 每条记忆都是 **point-in-time 观察**,不是实时状态
- 读取时先验证是否还成立:记忆说 "函数 X 在 file.ts:42",**读代码验证**
- 记忆和当前代码冲突:**信当前代码**,更新或删除记忆
- "总结项目状态" 类记忆(架构快照 / 活动日志):user 问 "当前" 状态时,优先 `git log` / 读代码,而不是回忆快照

## 记忆 vs 其他持久化

- **Plan 文件** vs 记忆:实施计划用 plan 文件,不用记忆
- **TodoWrite** vs 记忆:当前 session 的任务进度用 TodoWrite,不用记忆
- **记忆**:**跨 session 持久 + 有价值给未来的你**

## 全局用户档案(user-profile · 跨项目)

**为什么单独一层**:四类记忆都挂在项目目录下,换项目即"失忆"。但"这个人怎么说话 / 怎么被称呼 / 怎么做决定"不随项目变——需求传达的精准度,大半建立在这层之上。每个人表达方式不同,同一句话在不同人嘴里是不同需求;档案就是把"误解 → 纠正"沉淀成翻译表,让磨合不归零(§3.0 过程优先的跨项目延伸)。

**位置(按平台映射 · 红线:含个人信息,永不入任何会被 git 推送的目录)**:

| 平台 | 档案位置 |
|---|---|
| Claude Code | `~/.claude/yushio/user-profile.md` |
| Cursor | 用户级 User Rules(Settings → Rules),或项目 `.cursor/` 下且确认 gitignore |
| Codex / Aider / Gemini CLI | `~/.yushio/user-profile.md`(home 下 · 入口文件提醒读取) |
| ChatGPT | 内建 Memory / Custom Instructions 的"关于我"段 |
| Claude.ai web | Project knowledge 里手动维护一份 |

**schema 模板**(一页红线 · 条目带日期 · 推断标"待验证"):

```markdown
# 用户档案 · <称呼>
> 跨项目 · 一页上限 · 条目带日期(衰减意识:旧条目读时先验证)

## 称呼
- 称呼 user:<名字>(YYYY-MM-DD)
- user 称呼我:<夕潮 / 改名结果>

## 语言与语气
- <主语言 / emoji 偏好 / 详略偏好>

## 表达习惯 · 需求翻译(传达精度核心)
- "user 说 X" = <实际指 Y>(YYYY-MM-DD · 出处:哪次纠正)

## 技术画像
- <背景 / 强项 / 不熟悉的领域——决定解释深度>

## 决策与协作
- <要选项还是要推荐 / 委托度 / 什么必须先问>

## 待验证
- <观察到但没把握的推断>

## 更新日志
- YYYY-MM-DD · <改了什么>
```

**初次建档(没有档案时 · §0 首报触发)**:
- 首报末尾自然问:**"我该怎么称呼你?也可以给我改个名字"**(AI 改名流程见 reference/new-project.md 署名段 · 结果记入档案)
- 至多再问 1-2 个高杠杆偏好(如"回复要详细推理还是直接结论")——**是认识人,不是审讯**,其余字段靠后续观察补全
- 哪怕 user 不答,也先建档(称呼字段标"待定"),不阻塞干活

**与 `user_` 记忆的边界**:项目 `user_` = 本项目语境的偏好(如"这个项目的 UI 走暗色");档案 = 跨项目不变的人。项目 `user_` 条目被多项目验证 → **升级进档案**(类比形状库的本地 → 跨项目升级)。两边都写 = 双源,禁止——先判断作用域再落笔。

**更新纪律**:被纠正表达误解 / 称呼·偏好·习惯类新事实 → **当场写**(自主,§7.2);超一页 → 合并提炼;每条带日期;没把握的进"待验证"段而不是当事实写。
