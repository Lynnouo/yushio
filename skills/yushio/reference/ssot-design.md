# SSOT 设计纪律 · 把你最擅长的那层外化成机器可读的单一真相源

> 这是基础夕潮 §4.2（验证锚点）和美术总监 §6.2（一致性巡检）的伴随参考文件，不是独立 skill。
> 它回答一个问题：**当 user 不擅长写代码，却最懂数值 / 规则 / 视觉时，怎么让他在自己擅长的层上操作，而代码只是去读它？**
> 核心来自一个实证案例（某 React + TS 卡牌游戏项目 · 非程序员策划主导 · 用 CSV + 设计文档 + token 当真相源，快速做出一款完整游戏）。

---

## §1 为什么 SSOT 设计是"非程序员 + AI"的最大杠杆

基础夕潮 §3.0 讲过一个困境：大部分人"没有下达精准指令的能力"。SSOT 设计是这个困境的结构性解法——

> **不是让 user 学会写代码，而是把 user 真正擅长的那一层，外化成一份机器可读的单一真相源（SSOT）。user 在 SSOT 上操作（他的母语），代码只负责读它。**

- 游戏策划的母语是**数值 + 规则表** → SSOT = CSV / 配置表
- 设计师的母语是**视觉语言** → SSOT = design token / palette
- 产品的母语是**业务规则 + 流程** → SSOT = 设计文档 / 状态机定义

杠杆在于：user 改 SSOT（他会的），不碰代码（他不会的）；AI 保证代码忠实地读 SSOT。两层在 SSOT 处对接。

## §2 判据 · 什么该进 SSOT

| 该进 SSOT | 不该进 SSOT（留在代码） |
|---|---|
| 业务数值（价格 / 概率 / 阈值 / 时长 / 奖励） | 实现细节（数组下标 / 循环初值 / 框架样板） |
| 业务规则（谁能解锁什么 / 状态转移） | 算法本身（排序 / 渲染管线） |
| 文案 / 翻译 / 标签 | 控制流 / 错误处理 |
| 视觉 token（颜色 / 间距 / 字号 / 动效时长） | 一次性的脚手架 |
| 实体定义（卡牌 / 商品 / 关卡的字段） | —— |

**铁律（基础夕潮 §4.6 的延伸）**：

- **数值不准代码硬编码**——业务规则类的魔法值必须在 SSOT 里（`grep -rnE "if.*[<>=]\s*[0-9]{2,}"` 自查）。
- **代码只读 SSOT，不写回**——SSOT 是单向真相源；运行时产生的状态走别处，不回写真相源（防止"真相"被运行时污染）。
- **一个事实只有一处**——同一个数值 / 规则 / 颜色不在两个地方各写一份（否则必然 drift，见 §3）。

## §3 Drift 防御 · SSOT 与代码偏离必须"会报错"

SSOT 最大的失败模式：**真相源说 A，代码实装成 B，而没有任何东西报错**（typecheck / lint / build 全过）。这是设计形状 **#DF（签字稿 hex 实装时静默漂移）** 的根：作者 + 执行者同人时，自觉记忆必然漂移，**只有"机器会报错"是真防御**。

四层防御（从硬到软）：

1. **代码 SSOT 常量**：SSOT 值同步为代码可引用的常量（`design-tokens/*.ts` / `getConfig(key)`），不让代码里散落字面量。
2. **校验脚本 / hook**：写 `verify:<domain>` 脚本，正则提取 SSOT 值 vs 代码值双向对比，不一致 `exit 1`，挂进 pre-commit / CI（如 `validate-csv.sh`）。
3. **消费端 lint**：grep 组件 / 业务代码里"不在 SSOT 表内的字面量"（内联 hex / 魔法数），阻止 commit。
4. **交接核验**：跨 session / 跨 sprint 交接时，逐项对照 SSOT 与实装，偏离项标 `⚠️ 偏离真源`。

**判定**：SSOT 里有明确的值清单时 → 问两个问题——

- 代码里有对应的 SSOT 常量 / 查询入口吗？
- **偏离 SSOT 会报错吗**（CI / lint / test 任一）？

两个答"不"= drift 风险高，先建防御再开工。

## §4 实证案例（某 React + TS 卡牌游戏项目 · 非程序员主导）

- **CSV 即真相源**：一批业务 CSV（数值 / 实体 / 关卡）+ 一张 `GameConfig.csv`（全局常量 K/V）。策划改 CSV = 改游戏，不碰代码。
- **铁律落地**：`CLAUDE.md`「数值不准代码硬编码」+ `.claude/rules/csv-files.md` 锁格式 + `registry-schema.md` 锁外键矩阵。
- **机器护栏**：`validate-csv.sh` hook 在提交时校验 CSV 格式 / 外键 / 单位标度，违规即失败。
- **drift 防御**：`registry-schema.md` 外键矩阵 + 一个一致性巡检 skill（CSV × 代码 × 资源 × i18n 四联对账）。
- **设计 token 同源**：主题 palette 文件（颜色 token 集中处）是颜色 SSOT，「禁止硬编码颜色」铁律 + 稀有度配置与颜色镜像同步注释。

## §5 引用关系

- **基础夕潮 §4.2**：每个新项目第一个 session 回答验证锚点表时，**同时设计 SSOT**——"我擅长的哪层该外化成机器可读真相源？代码怎么只读它？"
- **美术总监 §6.2**：一致性巡检时，palette / token 是否有代码 SSOT + 偏离是否会报错（本文件 §3 + 形状 #DF）。
- **并行夕潮 §3**：SSOT / 稳定契约是"减小共享脊柱"的关键手段——能下沉到 SSOT 的就下沉，脊柱越小越好并行。
- **形状库**：#DF（hex 静默漂移）、#DG（视觉模式漂移）是 SSOT drift 的设计层实例；#DK（陈旧产物）是 SSOT 没清干净的实例。
